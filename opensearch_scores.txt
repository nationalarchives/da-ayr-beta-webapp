OPENSEARCH SCORE DEMONSTRATION

 Current Dynamic Boost Settings:

  Sort by "file_name" (default):
  - file_name*3 - 3x boost
  - All other fields: standard weighting

  Sort by "description":
  - description*3 - 3x boost
  - file_name*2 - 2x boost
  - Other fields: standard weighting

  Sort by "metadata":
  - file_name*0.2 - Reduced to 20%
  - content*0.1 - Reduced to 10%
  - Other metadata fields: standard weighting

  Sort by "record":
  - content*3 - 3x boost
  - file_name*2 - 2x boost

  Sort by "least_matches" & "most_matches":
  - No boosts applied (all fields equal)


  Current fuzziness settings:
  for term in single_terms:
      must_clauses.append(
          {
              "multi_match": {
                  "query": term,
                  "fields": search_fields,  # All fields get same treatment
                  "fuzziness": "AUTO",      # Hardcoded AUTO for everything
                  "lenient": True,
              }
          }
      )

Boost Points:
  - Default behavior (sort="file_name") gives file_name 3x - this is where the .msg ‚Üí MIG/Ms problem is amplified
  - Dynamic boosting changes based on user's sort preference - file name matches get 3x boost when
  sorting by file name
  - No fuzzy configuration in this function - it only handles field boosts, the fuzziness: 'AUTO' is applied
  elsewhere
  - The 3x boost on file_name makes fuzzy false matches in filenames score much higher than exact matches in
  content, which is why MIG Aircraft Manual.txt scores higher than content matches when searching .msg

Fuzziness Points:
 - Every single term search uses fuzziness: "AUTO"
 - All fields (file_name, description, content, etc.) get the same fuzzy settings
 - No field-specific configuration - file extensions get same fuzzy searching as natural language content
 - No ability to use exact matching for specific field types


SEARCH TYPES TESTED:
File extensions: .msg .pdf .txt .doc .xlsx | Extension searches: msg pdf txt doc xlsx
False match targets: MIG Ms FOI REF TDR | Common terms: file document report email january ministry
Typo variations: fil flie ministy januray | Short terms: A B C IT HR Fi
Date references: Monday Jan Mon 2023-01-01 | Year tests: 2024 2023 2022 2021 2020

FUZZINESS SETTINGS USED:
LOOSE (Problem): fuzziness 'AUTO', max_expansions 100 (all fields)
VERY LOOSE (Worse): fuzziness '2', max_expansions 200 (all fields)
EXACT: fuzziness '0' (all fields)
SLIGHTLY FUZZY: filename fuzziness '0', content fuzziness 'AUTO', max_expansions 50
CONSERVATIVE: filename fuzziness '1' + prefix_length 1, content fuzziness 'AUTO'
PREFIX PROTECTED: fuzziness 'AUTO' + prefix_length 2 (filename), prefix_length 1 (description)




PROBLEMS IDENTIFIED (BEFORE Section):

  Current Loose (Problem): fuzziness 'AUTO', max_expansions 100
  - Result: Massive false matching
  - .msg searches return MIG Aircraft Manual.txt and Ms Letter.txt with high scores
  - Short terms like A, B, C over-match everything
  - Year searches like 2024 incorrectly match 2023, 2022
  documents

  Very Loose (Worse): fuzziness '2', max_expansions 200
  - Result: Even more over-matching
  - Nearly every search returns irrelevant results
  - Complete loss of search precision

  SOLUTIONS FOUND (AFTER Section):

  1. Slightly Fuzzy

  Settings: filename fuzziness '0', content fuzziness 'AUTO', max_expansions 50

  Why chosen:
  - Prevents all false matches for file extensions (.msg only finds .msg files)
  - Maintains fuzzy search for natural language in descriptions/content
  - Field-specific approach - exact where precision matters, fuzzy where flexibility helps
  - Balanced performance with moderate max_expansions

  2. Conservative Fuzzy

  Settings: filename fuzziness '1' + prefix_length 1, content fuzziness 'AUTO'

  Why chosen:
  - Allows minor typos in filenames (fil ‚Üí file)
  - Prefix protection prevents distant matches (january ‚â† canary)
  - Still some risk of false matches but much reduced
  - Good compromise between precision and typo tolerance

  3. Prefix Protected

  Settings: fuzziness 'AUTO' + prefix_length 2 (filename), prefix_length 1 (description)

  Why chosen:
  - Forces first characters to match exactly
  - Prevents distant false matches
  - Maintains AUTO fuzziness for legitimate typos
  - May be too restrictive for some legitimate searches

  4. Exact Match Only

  Settings: fuzziness '0' (all fields)

  Why chosen:
  - Zero false positives - completely eliminates the problem
  - Perfect precision for critical searches
  - Too restrictive for most users (no typo correction)
  - Good for reference codes and exact term searches

  KEY INSIGHTS:

  1. Slightly Fuzzy is optimal because it solves the core problem (file extension false matches)
  while maintaining search flexibility for content
  2. Field-specific configuration is essential - one size doesn't fit all search scenarios
  3. Prefix protection works but may be too restrictive for natural language
  4. Current AUTO settings cause demonstrable problems with concrete examples in the test data
  5. The 3x filename boost amplifies the false matching problem - fuzzy matches in filenames score higher than
  exact matches in content

  The analysis shows that the Slightly Fuzzy (exact for filenames, AUTO for content) might perform better
